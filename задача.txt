ЧАСТЬ 2. ОБЩИЕ ПРИНЦИПЫ АРХИТЕКТУРЫ

Приложение — только 1:1 мессенджер (MVP).

Архитектура должна позволять добавить группы позже без переписывания логики.

Идентификация пользователя — только по user_id (UUID).

Нельзя использовать @username для связей.

Realtime:

Основной механизм — broadcast.

Fallback — подписка на insert + синхронизация при reconnect.

Offline:

Нет очереди сообщений.

При отсутствии сети — показывается баннер.

Отправка блокируется или возвращает ошибку.

ЧАСТЬ 3. ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ
3.1 Поля
@username

Уникальный.

Разрешены только символы: a-z.

Только lowercase.

Regex: ^[a-z]+$

Можно менять неограниченно.

Перед сохранением обязательно проверять уникальность.

display_name

Не уникальный.

Может быть пустым.

Обязательная санитизация (экранирование HTML).

avatar

Хранится только текущий.

История не нужна.

3.2 Правила отображения

Имя:

Если display_name не пуст → показывать его.

Иначе показывать @username.

Иначе “User”.

Аватар:

Если avatar есть → показывать.

Иначе:

инициалы display_name

если пуст → инициалы @username

ЧАСТЬ 4. ПОИСК И СОЗДАНИЕ ДИАЛОГА
4.1 Поиск пользователей

Поиск осуществляется:

по @username (уникальный)

по display_name (неуникальный)

Карточка результата:

avatar

display_name

@username

4.2 Создание диалога

Писать можно всем.

Диалог создаётся только при первом отправленном сообщении.

В системе должен существовать ровно один диалог на одну пару пользователей.

Нельзя создавать дубликаты.

ЧАСТЬ 5. СООБЩЕНИЯ
5.1 Типы

text

image

audio

5.2 Фото

Только из галереи.

До 4 за одну отправку.

Каждое фото — отдельное сообщение.

Обязательно клиентское сжатие.

Лимит: ~10MB на файл.

5.3 Голосовые

Запись внутри приложения.

Максимум 2 минуты.

Обязателен preview:

прослушать

удалить

перезаписать

Лимит: ~10MB.

5.4 Медиа

Загружается в storage.

В сообщении хранится:

URL

тип

размер

длительность (если audio)

ЧАСТЬ 6. СТАТУСЫ СООБЩЕНИЙ

У каждого исходящего сообщения отображается статус:

"Отправляется"

Локально до подтверждения.

"Отправлено"

После успешного insert.

"Прочитано"

Когда собеседник обновил last_read_at.

Правила:

Пользователь не может прочитать сам своё сообщение.

Разрешён внутренний статус "failed" с кнопкой Retry.

ЧАСТЬ 7. READ RECEIPTS

Механизм:

per-conversation last_read_at.

Правило (гибрид C):

Если пользователь открыл чат и уже находится внизу → обновить last_read_at.

Если не внизу → обновить только после скролла вниз.

Непрочитанные:

messages.created_at > last_read_at

ЧАСТЬ 8. REALTIME
8.1 Новые сообщения

Основной механизм: broadcast.

Fallback:

подписка на insert

синхронизация при reconnect.

8.2 Typing

typing:start при первом вводе.

keep-alive пока печатает.

typing:stop через 3 секунды без событий.

UI: “печатает…”

8.3 Voice recording indicator

recording:start

recording:stop

UI: “записывает голосовое…”

ЧАСТЬ 9. PRESENCE

Пользователь online пока приложение активно.

В списке чатов показывать online-индикатор.

ЧАСТЬ 10. PUSH-УВЕДОМЛЕНИЯ

Push отправляется:

на любое входящее сообщение

даже если чат открыт

не отправляется на свои сообщения

Текст:

Текст: "Новое сообщение от {name}: {text}"

Фото: "Новое сообщение от {name}: Фото"

Аудио: "Новое сообщение от {name}: Голосовое сообщение"

Звук и вибрация обязательны.
