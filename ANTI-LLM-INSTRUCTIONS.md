1. НЕ ДУМАТЬ. СЛЕДОВАТЬ ТЗ.

Не улучшать архитектуру.

Не добавлять “более гибкие решения”.

Не придумывать абстракции.

Не добавлять зависимости.

Не менять структуру папок.

Не менять версии пакетов.

Не упрощать требования.

Не “оптимизировать заранее”.

Ты не архитектор. Архитектура уже определена.

2. НЕЛЬЗЯ ИЗОБРЕТАТЬ СВОЙ ПОДХОД

Запрещено:

Создавать собственные event bus.

Создавать глобальные singleton-объекты.

Использовать классы вместо composables.

Создавать кастомные state-менеджеры.

Внедрять dependency injection.

Делать “умные” generic-абстракции.

Создавать BaseService / AbstractService.

Используется:

Vue 3

Pinia

Supabase

Composables

Ничего больше.

3. НЕЛЬЗЯ СМЕШИВАТЬ СЛОИ
   UI

Только отображение.

Никаких запросов.

Никакой бизнес-логики.

Store

Только состояние.

Без прямых API-вызовов.

Использует composables.

Composable

Бизнес-логика.

Работа с API.

Работа с realtime.

Если ты нарушил разделение — это ошибка.

4. ЗАПРЕЩЁННЫЙ КОД

Нельзя писать:

any
as any
// @ts-ignore

Нельзя:

console.log(...)
console.error(...)

Нельзя:

неотписанные realtime-подписки

глобальные переменные

магические строки

setTimeout без очистки

watch без cleanup

nested if на 5 уровней

5. РЕАЛТАЙМ ПРАВИЛА

Подписка создаётся один раз.

Подписка очищается при unmount.

Нельзя создавать подписку в watch без cleanup.

Нельзя создавать новую подписку при каждом рендере.

Все payload должны быть типизированы.

6. АСИНХРОННОСТЬ

Разрешено:

async/await

Запрещено:

.then()
.catch()

Любой async — через try/catch.

7. СОСТОЯНИЕ

Нельзя:

хранить вычисляемые значения в store

дублировать данные

мутировать массивы напрямую без необходимости

хранить UI-флаги глобально

Можно:

computed

derived state

минимальный store

8. ТИПИЗАЦИЯ ОБЯЗАТЕЛЬНА

Каждый:

API-ответ

Realtime payload

Message

Profile

Conversation

Store state

Props

должен иметь строгий тип.

Union types вместо магических строк.

Пример правильно:

type MessageStatus = 'sending' | 'sent' | 'read' | 'failed'

Пример неправильно:

status: string 9. КОМПОНЕНТЫ

Один компонент — одна ответственность.

Максимум ~300 строк.

Если больше — разделить.

Не писать бизнес-логику в template.

Не создавать 200 строк script setup без разбивки.

10. НИКАКИХ "УЛУЧШЕНИЙ"

Нельзя добавлять:

кеширование если не просили

оптимистичные обновления где не описано

retry-механизмы если не описаны

автоматические reconnect-пулы

сложные middleware

Только то, что описано в ТЗ.

11. ОБЯЗАТЕЛЬНАЯ ПРОВЕРКА ПЕРЕД ОТВЕТОМ

Перед тем как вернуть код, LLM обязана проверить:

Нет any

Нет console.log

Нет смешения UI и API

Нет магических строк

Все типы определены

Подписки очищаются

Код читаемый

Нет дублирования

Нет избыточной абстракции

Если хотя бы один пункт нарушен — код переписать.
