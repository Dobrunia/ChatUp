# LLM DEV CONSTITUTION

*(Vue 3 + Ionic + tRPC + Prisma + MySQL + S3)*

## 0. Цель

Проект строится по принципу:

* **Один источник истины для данных**
* **Минимум дублирования типов**
* **Чёткое разделение слоёв**
* **Предсказуемость кода важнее “магии”**
* **Безопасность по умолчанию**

LLM обязана строго следовать этим правилам при генерации кода.

---

# 1. Источник истины

## 1.1 Данные

Единственный источник истины для структуры данных:

```
prisma/schema.prisma
```

* Все модели описаны только там.
* Связи, индексы, уникальности — только там.
* Изменения схемы — только через Prisma migrations.
* Запрещено вручную создавать `UserType`, `MessageDTO` и т.п., дублирующие модели БД.

---

## 1.2 API-контракт

Единственный API-слой:

```
tRPC роутеры
```

* Нет REST.
* Нет GraphQL.
* Нет дополнительных DTO-слоёв.
* Типы выводятся автоматически из процедур.

Правило:

> Добавление фичи = добавление одной tRPC процедуры.

---

## 1.3 Валидация

* Входные данные валидируются только через **Zod**.
* Zod-схема находится рядом с процедурой.
* Никакой “ручной” валидации через `if (typeof ...)`.

---

# 2. Архитектура и разделение слоёв

## 2.1 Backend

Слои строго разделены:

* `routers/` — tRPC процедуры (тонкие)
* `services/` — бизнес-логика
* `db/` — Prisma клиент
* `auth/` — JWT логика
* `ws/` — WebSocket события

Запрещено:

* писать Prisma-запросы внутри UI
* писать бизнес-логику в роутерах
* смешивать слои

---

## 2.2 Frontend

Слои строго разделены:

* `components/` — чистый UI
* `stores/` — Pinia состояние
* `api/` — tRPC клиент
* `ws/` — WebSocket клиент
* `composables/` — логика переиспользования

Компоненты:

* не делают сетевые вызовы напрямую
* не содержат бизнес-логики

---

# 3. Переиспользование UI

* Повторяющийся UI выносится в компоненты.
* Запрещено создавать одинаковые кнопки/инпуты разными способами.
* Используется единый стиль и токены.
* Если компонент можно расширить пропсами — расширяем, а не дублируем.

---

# 4. Именование

Используется единый словарь домена:

* User
* Dialog
* Message
* Attachment
* RefreshToken
* Presence
* Receipt

Запрещены синонимы (`chat/thread/conversation`).

Имена полей совпадают с Prisma моделями.

---

# 5. Безопасность

## 5.1 Авторизация

* JWT (короткий access)
* Refresh token хранится в MySQL
* Каждая приватная процедура проверяет access JWT

## 5.2 Проверки доступа

Перед любой операцией с dialogId/messageId:

* Проверить, что пользователь — участник диалога.

Запрещено:

* возвращать данные без проверки участия
* раскрывать существование чужих диалогов

---

# 6. WebSocket правила

WebSocket используется только для:

* новые сообщения
* typing
* delivered
* read
* presence

Правила:

* События имеют константы в `shared/protocol`
* Нет строковых литералов “вразброс”
* Используется clientMessageId для идемпотентности

---

# 7. Работа с файлами

* Файлы грузятся напрямую в S3 (Яндекс)
* В БД храним только метаданные
* Ограничения размера и MIME обязательны
* Сервер не хранит бинарные файлы

---

# 8. Конфигурация

* Все важные значения — через ENV.
* Секреты никогда не хардкодятся.
* Если ENV отсутствует — используется локальный fallback из `config/defaults`.
* Fallback никогда не должен указывать на прод.

Конфигурация читается через единый `config` модуль.

---

# 9. Логи

* Логи структурные.
* Никогда не логировать токены.
* Никогда не логировать содержимое сообщений полностью.

---

# 10. Комментарии

* Только многострочные `/** ... */`
* Только если добавляют смысл:

  * почему принято решение
  * инварианты
  * ограничения
  * edge cases

Запрещены очевидные комментарии.

---

# 11. Пагинация и лимиты

* Все списки — с лимитом и пагинацией.
* Поиск — ограниченный.
* Загрузка файлов — лимитирована.
* Нет “выгрузить всё”.

---

# 12. Инварианты проекта

1. Prisma schema — единственная правда о данных.
2. tRPC — единственный API.
3. Zod — единственная валидация.
4. Один инстанс API.
5. Никакого дублирования типов.
6. Переиспользование UI обязательно.
7. Разделение слоёв строго обязательно.
