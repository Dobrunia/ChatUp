# CODE_STANDARDS.md

## Правила написания аккуратного и поддерживаемого кода

Этот документ обязателен к соблюдению.
Нарушение правил считается ошибкой реализации.

---

# 1. Главные принципы

1. Код должен быть:

   * предсказуемым
   * типизированным
   * разделённым по слоям
   * легко читаемым

2. Никакой “магии”.

3. Никаких скрытых сайд-эффектов.

4. Никаких спонтанных архитектурных решений.

5. Не придумывать абстракции заранее.

---

# 2. Структура проекта

Обязательная структура:

```
src/
  app/
    router/
    providers/
  modules/
    auth/
    profile/
    conversations/
    chat/
    search/
  shared/
    api/
    composables/
    ui/
    types/
    utils/
  stores/
```

## Правила:

* Каждый модуль изолирован.
* Нельзя импортировать модуль в модуль циклически.
* shared не зависит от modules.
* stores не должны содержать UI-код.

---

# 3. Разделение ответственности

## 3.1 UI (Vue компоненты)

* Только отображение.
* Никаких запросов к API.
* Никакой бизнес-логики.
* Вся логика — через composables или store.

## 3.2 Composables

* Бизнес-логика.
* Работа с realtime.
* Работа с network.
* Без разметки.

## 3.3 Stores (Pinia)

* Хранят состояние.
* Не содержат HTTP-запросы напрямую.
* Используют composables.

## 3.4 API слой

* Только вызовы Supabase.
* Без логики UI.
* Без форматирования данных для отображения.

---

# 4. TypeScript — строгая дисциплина

1. Строгий режим обязателен.
2. any запрещён.
3. Все API-ответы должны иметь тип.
4. Все payload realtime должны иметь тип.
5. Все props компонентов должны быть типизированы.

Пример:

```ts
type MessageType = 'text' | 'image' | 'audio'

interface Message {
  id: string
  conversationId: string
  senderId: string
  type: MessageType
  body?: string
  media?: MediaMeta
  createdAt: string
}
```

---

# 5. Работа с состоянием

1. Состояние должно быть:

   * минимальным
   * нормализованным
   * без дублирования

2. Нельзя хранить:

   * вычисляемые данные (использовать computed)
   * UI-состояние в глобальном store (если оно локальное)

3. Не мутировать объекты без необходимости.

---

# 6. Realtime дисциплина

1. Подписки должны:

   * отписываться при unmount
   * не создаваться повторно без очистки

2. Нельзя:

   * создавать подписку внутри watch без cleanup
   * создавать подписки в каждом рендере

3. Все realtime события должны быть:

   * строго типизированы
   * обработаны через единый composable

---

# 7. Работа с ошибками

1. Нельзя игнорировать ошибки.

2. Любой async должен быть обёрнут в try/catch.

3. UI должен получать понятное состояние:

   * loading
   * success
   * error

4. console.log запрещён в production-коде.

---

# 8. Асинхронность

1. Использовать async/await.
2. Не смешивать then/catch.
3. Не создавать вложенные промисы.
4. Не использовать setTimeout для логики (кроме typing таймаута).

---

# 9. Нейминг

1. Файлы:

   * kebab-case
2. Компоненты:

   * PascalCase
3. Переменные:

   * camelCase
4. Типы:

   * PascalCase
5. Константы:

   * UPPER_SNAKE_CASE

Плохой пример:

```
data.js
helper.ts
utils2.ts
```

Хороший пример:

```
chat-realtime.ts
useChatMessages.ts
profile-validator.ts
```

---

# 10. Компоненты

1. Один компонент — одна ответственность.
2. Не более ~300 строк.
3. Если больше — разбивать.
4. Не вкладывать бизнес-логику в template.

---

# 11. Форматирование

1. Prettier обязателен.
2. ESLint без ошибок.
3. Импорты отсортированы.
4. Не оставлять закомментированный код.

---

# 12. Безопасность

1. Всегда санитизировать display_name.
2. Никогда не доверять данным из БД.
3. Проверять тип message.type перед рендером.
4. Не рендерить HTML из строк.

---

# 13. Медиа

1. Прогресс загрузки обязателен.
2. Не блокировать UI во время загрузки.
3. Обязательно обрабатывать ошибки загрузки.

---

# 14. Push и Network

1. Проверять статус сети через Capacitor Network.
2. Не отправлять сообщение если offline.
3. Обрабатывать ошибки push-регистрации.

---

# 15. Антипаттерны (запрещено)

* Божественные компоненты (God Component).
* Store на 2000 строк.
* Копипаста логики.
* Inline-функции на 50 строк.
* Глубокая вложенность if.
* Использование any.
* Магические строки без enum/union types.
* Логика в mounted без необходимости.

---

# 16. Что считается “senior-подходом”

* Малые функции.
* Чёткие границы модулей.
* Предсказуемые данные.
* Отсутствие скрытых зависимостей.
* Минимум состояния.
* Прозрачный realtime lifecycle.
* Типы описывают систему лучше комментариев.
