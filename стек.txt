Стек: бэкенд (минимум, но правильно)
Цели

1 источник истины для типов/данных → БД-модели описаны один раз

Минимум ручных типов/DTO → типы автоматически “протекают” на фронт

Один инстанс API (один серверный процесс)

1) Единственный источник истины
✅ prisma/schema.prisma — единственная правда о данных

Все сущности (User, Dialog, Message, Attachment, RefreshToken и т.п.)

Связи, индексы, уникальности

Prisma миграции

Правило: никаких ручных types.ts с моделями User/Message и никаких дублей структур “как в БД”.

2) Сервер (API)
✅ Node.js + tRPC + Prisma + Zod

tRPC — один слой API (процедуры), без REST/GraphQL схем

Prisma Client — доступ к MySQL строго по моделям Prisma

Zod — единственная валидация входных данных (input) прямо в процедурах tRPC

Как добавлять новую фичу (без бойлерплейта):

добавляешь одну tRPC процедуру

описываешь input через Zod

пишешь запрос через Prisma

фронт сразу получает типы автоматически

Правило: не писать отдельные DTO/Response типы. Выходные типы выводятся из Prisma/tRPC.

3) База данных
✅ MySQL 8

Prisma работает с MySQL напрямую

Структура и типы — строго из Prisma schema

Правило: любые изменения схемы — только через Prisma migrations.

4) Realtime для чата
✅ WebSocket слой (для сообщений/typing/ack/read)

Отдельный WS-канал рядом с API (в том же приложении)

tRPC используешь для: auth, профиль, поиск, список диалогов, история сообщений

WS используешь для: новые сообщения и статусы доставки/прочтения

(Один инстанс API → можно держать подключения прямо в процессе.)

5) Хранение файлов
✅ S3 от Яндекса

Клиент грузит файлы в S3

Сервер хранит в MySQL только метаданные (key, size, mime, owner, dialog/message link)

6) Auth
✅ JWT + Refresh token

Access JWT короткий (минуты)

Refresh токен длинный

Refresh не через httpOnly cookie, а явным токеном (например, в secure storage на мобилке)

Refresh токены храним в MySQL (как записи, чтобы можно было отзывать)

Правило: каждая tRPC процедура (кроме публичных) требует проверку access JWT.

7) Контракт для LLM (чтобы не путалась)

Prisma schema = модели и связи (единственный источник истины по данным)

tRPC роутеры = единственный API контракт

Zod input рядом с процедурой = единственный источник истины для входных данных

Фронт получает типы из tRPC (без отдельного ручного слоя типов)

Дополнительные правила разработки (обязательные)

Конфигурация и ENV

Все важные значения (URL API, ключи JWT, DSN MySQL, доступы к S3 Яндекса, лимиты файлов, таймауты, домены) задаются только через ENV.

В коде не хранить секреты и прод-настройки.

Для локальной разработки допускается fallback: если переменная ENV отсутствует или пустая — подставляем безопасное значение по умолчанию из config/defaults (только для локального тестирования). При этом:

значения по умолчанию не должны позволять случайно подключиться к прод-ресурсам;

в лог при старте выводим явное предупреждение, что используется fallback-конфиг.

Конфигурация читается через единый модуль config (одна точка входа), а не через process.env по всему проекту.

Переиспользование UI и единый стиль

Любые повторяющиеся элементы UI (кнопки, инпуты, хедеры, карточки, аватары, элементы списка чатов/сообщений, модалки, тосты) обязательно выносим в компоненты и переиспользуем.

Запрещено создавать “10 разных реализаций одной и той же кнопки/ячейки” — сначала ищем существующий компонент, и только если его реально не хватает, расширяем его пропсами.

Для визуальной консистентности: один источник стилей и токенов (например, общий файл с константами размеров/отступов/вариантов).

Разделение по слоям — обязательно

UI-слой (компоненты) не содержит бизнес-логики и прямых запросов к сети.

Бизнес-логика и состояние — в stores/services (Pinia/composables).

Сетевой слой — отдельный модуль (api/trpcClient, ws/client) без “быстрых fetch внутри компонентов”.

На бэкенде: tRPC роутеры тонкие, бизнес-логика выносится в services, доступ к БД — через repositories/Prisma-клиент (единый слой), без SQL/Prisma вызовов, размазанных по проекту.

Комментарии

Комментарии пишем только многострочные (/** ... */) и только там, где они добавляют смысл:

почему принято решение,

какие инварианты важны,

какие edge cases учтены,

что нельзя ломать.

Запрещены “очевидные” комментарии типа // set user — вместо этого код должен быть самодокументируемым через имена и структуру.
1) Именование и единый словарь домена

Везде используем одни и те же термины: User, Dialog, Message, Attachment, Presence, Receipt.

Не плодим синонимы (chat/thread/conversation) — выбираем один термин и держимся его.

Имена полей строго совпадают с Prisma моделями (где возможно), чтобы не было “messageId vs idMessage”.

2) Жёсткие инварианты и проверки доступа

Любая операция с dialogId/messageId обязана начинаться с проверки: пользователь участник диалога.

Запрещены методы “получить сообщение по id” без контекста диалога/участия.

Ошибки доступа возвращаем одинаково (без утечек “существует ли диалог/пользователь”).

3) Единый формат ошибок

Все ошибки API — через единый helper (например throwAppError(code, message, meta)).

На клиенте — единый маппер ошибок в UI (тосты/диалоги), без “ловим ошибку по-разному в каждом компоненте”.

4) Идемпотентность и стабильные идентификаторы

Для отправки сообщений используем clientMessageId (UUID), чтобы повторная отправка при плохой сети не создавала дубликаты.

Сервер должен уметь ответить “это сообщение уже есть” по clientMessageId.

5) Лимиты и защита от “взрыва”

Все списки (диалоги/сообщения/поиск) только с пагинацией и лимитами.

Любая загрузка файлов — лимит размера + лимит типов + лимит количества вложений на сообщение.

Поиск пользователей — лимит частоты и минимум данных в ответе.

6) Одна точка правды для “констант протокола”

Все event names для WS, статусы доставок, лимиты, таймауты — в одном модуле shared/protocol.

На фронте и бэке используем эти константы, а не строки в коде.

7) Логи (структурные) и запрет логирования секретов

Логи только структурные ({ event, userId, dialogId, ... }).

Запрещено логировать токены, refresh, заголовки авторизации, содержимое сообщений целиком (особенно в проде).

8) Тестируемость без фанатизма

Минимум: тесты на критичные правила доступа (участник диалога / не участник) и на идемпотентную отправку.

Без “юнит-тестов на каждый setter”, но обязательны тесты на безопасность и корректность основных потоков.

9) Коммиты и структура PR (чтобы LLM не “размазала” изменения)

Одно изменение — один PR/ветка/коммит-серия.

Запрещены “рефакторинг + фича + форматирование” в одной пачке.

Любой рефакторинг — без изменения поведения (или отдельно описывать).

10) Запрет “умной магии” и скрытых зависимостей

Никаких неявных сайд-эффектов при импорте модулей (особенно в shared).

Инициализация (подключение к БД, поднятие WS) — строго в bootstrap/main.
